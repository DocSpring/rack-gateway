Rack Gateway uses a comprehensive testing strategy with unit tests, integration tests, and end-to-end tests across both Go and TypeScript codebases.

## Test Overview

| Test Type | Framework | Location | Command |
|-----------|-----------|----------|---------|
| Go Unit Tests | Go testing | `*_test.go` files | `task go:test` |
| Go Integration Tests | Go testing | `internal/integration/` | `task go:test` |
| Web Unit Tests | Vitest | `web/src/**/*.test.ts` | `task web:test` |
| Web E2E Tests | Playwright | `web/e2e/` | `task web:e2e` |
| CLI E2E Tests | Shell scripts | `config/cli-e2e/` | `task go:e2e` |

## Running Tests

### Full Test Suite

```bash
# Run ALL tests (recommended before committing)
task ci

# This runs:
# - Linting (Go + Web)
# - Go unit and integration tests
# - Web unit tests
# - E2E tests (Web + CLI)
# - Build verification
```

### Go Tests

```bash
# Run all Go tests
task go:test

# Run specific package tests
go test ./internal/gateway/auth/...

# Run with verbose output
go test -v ./internal/gateway/rbac/...

# Run specific test by name
go test -v -run TestRBACPolicy ./internal/gateway/rbac/
```

### Web Tests

```bash
# Run all web unit tests
task web:test

# Run in watch mode (development)
cd web && bunx vitest

# Run with UI
cd web && bunx vitest --ui

# Run specific test file
cd web && bunx vitest src/components/Button.test.tsx
```

### End-to-End Tests

```bash
# Run all E2E tests
task e2e

# Run web E2E only
task web:e2e

# Run CLI E2E only
task go:e2e

# Run specific Playwright test
cd web && bunx playwright test e2e/auth.spec.ts
```

## Writing Tests

### Go Unit Tests

```go
package auth_test

import (
    "testing"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
    "github.com/docspring/rack-gateway/internal/gateway/auth"
)

func TestSessionManager_CreateSession(t *testing.T) {
    // Arrange
    manager := auth.NewSessionManager(testConfig)
    user := &auth.User{Email: "test@example.com"}

    // Act
    session, err := manager.CreateSession(user)

    // Assert
    require.NoError(t, err)
    assert.NotEmpty(t, session.Token)
    assert.Equal(t, user.Email, session.UserEmail)
}

func TestSessionManager_ValidateSession(t *testing.T) {
    tests := []struct {
        name    string
        token   string
        wantErr bool
    }{
        {
            name:    "valid token",
            token:   validToken,
            wantErr: false,
        },
        {
            name:    "expired token",
            token:   expiredToken,
            wantErr: true,
        },
        {
            name:    "malformed token",
            token:   "invalid",
            wantErr: true,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            _, err := manager.ValidateSession(tt.token)
            if tt.wantErr {
                assert.Error(t, err)
            } else {
                assert.NoError(t, err)
            }
        })
    }
}
```

### Go Integration Tests

```go
// internal/integration/auth_test.go
package integration_test

import (
    "net/http"
    "testing"
    "github.com/stretchr/testify/suite"
)

type AuthIntegrationSuite struct {
    suite.Suite
    server *httptest.Server
    client *http.Client
}

func (s *AuthIntegrationSuite) SetupSuite() {
    // Start test server with real database
    s.server = setupTestServer(s.T())
    s.client = &http.Client{}
}

func (s *AuthIntegrationSuite) TearDownSuite() {
    s.server.Close()
}

func (s *AuthIntegrationSuite) TestOAuthFlow() {
    // Test complete OAuth authentication flow
    resp, err := s.client.Get(s.server.URL + "/api/v1/auth/login")
    s.Require().NoError(err)
    s.Equal(http.StatusOK, resp.StatusCode)
}

func TestAuthIntegration(t *testing.T) {
    suite.Run(t, new(AuthIntegrationSuite))
}
```

### Web Unit Tests (Vitest)

```typescript
// src/components/Button.test.tsx
import { describe, it, expect, vi } from 'vitest';
import { render, screen, fireEvent } from '@testing-library/react';
import { Button } from './Button';

describe('Button', () => {
  it('renders with correct text', () => {
    render(<Button>Click me</Button>);
    expect(screen.getByText('Click me')).toBeInTheDocument();
  });

  it('calls onClick when clicked', () => {
    const handleClick = vi.fn();
    render(<Button onClick={handleClick}>Click me</Button>);

    fireEvent.click(screen.getByText('Click me'));

    expect(handleClick).toHaveBeenCalledOnce();
  });

  it('is disabled when loading', () => {
    render(<Button loading>Click me</Button>);
    expect(screen.getByRole('button')).toBeDisabled();
  });
});
```

### Web E2E Tests (Playwright)

```typescript
// e2e/auth.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Authentication', () => {
  test('should redirect to login when unauthenticated', async ({ page }) => {
    await page.goto('/');
    await expect(page).toHaveURL(/.*login/);
  });

  test('should complete OAuth login flow', async ({ page }) => {
    await page.goto('/login');

    // Click login button
    await page.click('[data-testid="login-google"]');

    // Mock OAuth redirects to callback
    await expect(page).toHaveURL('/');
    await expect(page.locator('[data-testid="user-menu"]')).toBeVisible();
  });

  test('should show user email after login', async ({ page }) => {
    await loginAsUser(page, 'test@example.com');

    await page.click('[data-testid="user-menu"]');
    await expect(page.getByText('test@example.com')).toBeVisible();
  });
});
```

## Mocking

### Go Mocks

Use interfaces and mock implementations:

```go
// Define interface
type SessionStore interface {
    Get(token string) (*Session, error)
    Set(session *Session) error
    Delete(token string) error
}

// Mock implementation
type MockSessionStore struct {
    sessions map[string]*Session
}

func (m *MockSessionStore) Get(token string) (*Session, error) {
    if s, ok := m.sessions[token]; ok {
        return s, nil
    }
    return nil, ErrSessionNotFound
}
```

## Best Practices

### Do's

- Write tests alongside code (not after)
- Use table-driven tests for multiple cases
- Mock external services, not internal packages
- Keep tests fast (under 100ms per unit test)
- Test edge cases and error paths
- Use meaningful test names that describe behavior

### Don'ts

- Don't test implementation details
- Don't write flaky tests (no `time.Sleep`)
- Don't share state between tests
- Don't ignore test failures
- Don't skip tests without explanation
